# カリキュラム作成マニュアル

**1000円→10000円チャレンジ カリキュラムの品質基準とフィードバック改善プロセス**

---

## 🎯 このマニュアルの目的

1. **品質基準の明確化**: カリキュラムがどの水準に達すべきかを定義
2. **フィードバックループ**: ユーザーの「わからない」を改善につなげる
3. **継続的改善**: カリキュラムを段階的に洗練させる

---

## 🏗️ 骨子ファーストアプローチ（重要）

### Claude Codeの特性

**問題**: 一度に全部書くとクオリティ・密度が下がる

**理由**:
- 長文生成でコンテキストが薄れる
- 後半ほど簡略化される
- 細部への注意が散漫になる

**解決策**: 骨子を作ってから段階的に埋めていく

---

### ステップ1: 骨子（構造）を作成

```markdown
# Week X: タイトル

## 登場人物
[プレースホルダー]

## Part 1: 導入
### Scene 1: [シーン名]
[プレースホルダー]

## Part 2: [メイントピック]
### [歴史上の人物]の登場
[プレースホルダー]

## Part 3: 実践
[プレースホルダー]

## まとめ
[プレースホルダー]
```

**ポイント**:
- 全体の流れを決める
- 各Partの目的を明確化
- プレースホルダーで「何を書くか」をメモ

---

### ステップ2: 一項目ずつ埋めていく

**悪い例**（一度に全部）:
```
「Week 2を全部書いてください」
→ 最初は詳細、後半は簡略化
→ クオリティが不均一
```

**良い例**（段階的）:
```
手順:
1. 「登場人物のセクションを詳細に書いてください」
   → キャラクター設定を丁寧に作成

2. 「Part 1 Scene 1を詳細に書いてください」
   → 導入シーンに集中

3. 「ワイルダーの回想シーンを詳細に書いてください」
   → ストーリー性を高める

4. 「実践セクションを詳細に書いてください」
   → 技術的な内容を充実させる

→ 各セクションのクオリティが均一に高い
```

---

### ステップ3: レビューと洗練

各セクション完成後:
- [ ] ストーリー性はあるか？
- [ ] キャラクターは生き生きとしているか？
- [ ] 具体的な数字・状況はあるか？
- [ ] 自然な対話になっているか？

## 🎭 ストーリー性の強化

### 良いストーリーの要素

#### 1. 具体的なキャラクター設定

❌ **悪い例**（薄い）:
```markdown
**あなた（tatut）** - Week 1で初めての購入を経験
```

✅ **良い例**（具体的）:
```markdown
**ユウタ（23歳）** - バイト代8万円で生活する大学生
- 家賃: 5万円
- 食費: 2万円
- 光熱費: 1万円
- 残り: 雀の涙

「海外旅行とか行きたいのに、金がないんだよな...」

SNSで「100円が1000円になった！」という投稿を見て、
「100円なら...失っても痛くないし、やってみるか」と決意。
```

#### 2. リアルな状況設定

❌ **悪い例**（抽象的）:
```markdown
**ミコ**: 「購入結果を見てみよう」
```

✅ **良い例**（リアル）:
```markdown
**数日後、深夜1時、ユウタの部屋**

**ユウタ**: （スマホを見ながら）「はぁ...今日もバイト疲れた...」

**ユウタ**: 「そういえば、あの銘柄どうなったかな...」

**ユウタ**: （ダッシュボードを開く）「え、マジで!? +15%!?」

**ユウタ**: （興奮）「500円が575円になってる！75円も増えた！」

**ミコ**: （画面に現れる）「よお、ユウタ。儲かったみたいだな」
```

#### 3. 自然な対話

❌ **悪い例**（堅い）:
```markdown
**あなた**: 「...」
**ミコ**: 「RSIを確認しましょう」
**あなた**: 「わかった」
```

✅ **良い例**（自然）:
```markdown
**ユウタ**: 「ミコ、これヤバくない!? 75円も増えてる！」
**ミコ**: 「まあな。でも調子に乗るな。まずRSI見てみろ」
**ユウタ**: 「RSI...えーと、72.3？」
**ミコ**: 「そう。70超えてるだろ？Week 0で何習った？」
**ユウタ**: 「あ...買われすぎ...」
**ミコ**: 「そう。今利確するか、それとも持ち続けるか。お前が決めろ」
**ユウタ**: 「うーん...でもまだ上がるかも...」
**ミコ**: 「欲が出てるな。それが一番危ない」
```

#### 4. 感情の動き

```markdown
**心理描写を入れる**:

**ユウタ**: （内心）「まだ上がるかもしれない...でも、RSI 70超えてるし...」
**ユウタ**: （手が震える）「利確するか...？でも、もし明日もっと上がったら...」
**ユウタ**: （深呼吸）「...いや、ミコが言ってた。欲が出たら負けるって」
**ユウタ**: （決意）「利確する！」
```

## 👤 歴史上の人物を登場人物として活用

### なぜ歴史上の人物を使うのか？

1. **信頼性**: 実在の人物の苦労と成功
2. **共感**: 同じ失敗をしてきた
3. **教育的**: 発明の背景を理解できる

### 主要な人物

| 指標 | 発明者 | 年代 | 背景 |
|------|--------|------|------|
| **RSI** | J・ウェルズ・ワイルダー | 1978年 | 機械エンジニア出身、何度も負けて考案 |
| **MACD** | ジェラルド・アペル | 1970年代 | 投資顧問、トレンドフォローの限界に悩む |
| **BB** | ジョン・ボリンジャー | 1980年代 | テクニカルアナリスト、ボラティリティの測定 |

### 登場パターン

#### パターン1: 回想シーン

```markdown
### 【回想シーン: ワイルダーの悩み】

**場所**: ルイジアナ州ニューオーリンズ、自宅の書斎、1977年

**ワイルダー** (45歳、機械エンジニア出身):
「くそっ...また負けた」

*チャートを睨みつける。商品先物（大豆）のチャートだ。*

**ワイルダー**: 「価格が急騰してるから買った。そしたら翌日暴落...」
**ワイルダー**: 「なぜだ？なぜ俺が買った瞬間に下がる？」

*机の上には数ヶ月分のトレード記録。ほとんどが赤字。*
```

#### パターン2: 対話での言及

```markdown
**ミコ**: 「RSIがなぜこんなに有効か知ってるか？」
**ユウタ**: 「...なんで？」
**ミコ**: 「発明者の話をしよう。1978年、ニューオーリンズに住んでいた男、ワイルダーだ」
**ミコ**: 「彼も、お前と同じように何度も負けた」
**ユウタ**: 「...マジで？」
**ミコ**: 「ああ。でも彼は、そこから学んで、RSIを生み出した」
```

#### パターン3: 一人称視点（実験的）

```markdown
### ワイルダーの手記（1977年12月）

> 私は何度も負けてきた。
> 急騰する銘柄を買っては、翌日暴落する。
>
> しかし、ある日気づいた。
> 「急騰してる時に買うから負けるのではないか？」
>
> エンジンには「正常回転数」がある。
> それを超えるとオーバーヒートする。
>
> 株価も同じではないのか？
```

## 📊 カリキュラムの品質基準

### レベル1: 最低限（不合格）

❌ **これでは不十分**:
- 専門用語を説明なしで使っている
- 「なぜ？」に答えていない
- 抽象的な説明だけで具体例がない
- 初心者が疑問を持つポイントを無視している

**例（悪い）**:
```markdown
RSIは相対力指数です。70以上は買われすぎです。
```
- 「相対力指数」とは？
- なぜ70なのか？
- 買われすぎだと何がダメなのか？
→ すべて説明されていない

---

### レベル2: 基本（合格ライン）

✅ **最低限これが必要**:
- 専門用語を初心者向けに言い換えている
- 「なぜ？」の理由を1つ以上説明している
- 具体例が1つ以上ある
- 初心者の疑問を想定して先回りしている

**例（良い）**:
```markdown
RSI（相対力指数）= 「買われすぎ」「売られすぎ」を0-100の数値で示す指標

RSI 70以上 = 買われすぎ
→ なぜダメ？ → みんな買った → もう買う人がいない → 価格が下がる

例: BTC のRSI 75
→ 今は買わない方が良い。調整が入る可能性が高い
```

---

### レベル3: 高品質（理想）

🌟 **さらに良いカリキュラム**:
- 完全初心者（サヤ）が疑問を提起
- 一つずつ疑問を解消
- 何度も反芻（説明 → 質問 → 説明し返す → 確認）
- 実データでの例
- 失敗例と成功例の両方

**例（最高）**:
```markdown
**サヤ**: 「RSIって何の略？」
**ミコ**: 「Relative Strength Index。日本語で『相対力指数』」
**サヤ**: 「...相対力指数？」
**ミコ**: 「難しいな。簡単に言うと『買われすぎ・売られすぎメーター』だ」
**サヤ**: 「メーター？」
**ミコ**: 「そう。0から100の数字で表す。70以上なら『買われすぎ』」
**サヤ**: 「待って。買われすぎって...買われすぎたら何がダメなの？」
**ミコ**: 「いい質問だ。買われすぎってことは...？」
**サヤ**: 「みんな買ってる...もう買う人がいない...？」
**ミコ**: 「そう！買う人がいなくなったら、価格は...？」
**サヤ**: 「下がる！」
**ミコ**: 「正解。だから『RSI 70以上 = 調整（下落）の可能性』」

（実データで確認）
今のBTCのRSI: 75.2 🔴

**サヤ**: 「じゃあ今は買わない方がいい？」
**ミコ**: 「その通り。様子を見よう」
```

## 🔄 フィードバック改善プロセス

### ステップ1: ユーザーからのフィードバック受信

ユーザーが「わからない」と言った箇所を記録する。

**収集する情報**:
1. **どこが**わからなかったか（箇所）
2. **何が**わからなかったか（内容）
3. **なぜ**わからなかったか（理由）

**フィードバックの例**:
```
ユーザー: 「MACDのヒストグラムの意味がわからない」

記録:
- 箇所: Week 0 Part 3「MACDの見方」セクション
- 内容: Histogramの意味
- 理由: 「拡大」「縮小」が何を意味するのか説明不足
```

---

### ステップ2: 原因分析

**原因の分類**:

#### A. 説明不足
- 専門用語を説明していない
- 理由（なぜ？）を説明していない
- 具体例がない

#### B. 前提知識の欠如
- ユーザーが前提知識を持っていない
- Week 0で学ぶべきことをスキップしている

#### C. 構成の問題
- 情報の順番が悪い
- 関連する情報が離れている
- 重要な情報が埋もれている

#### D. 表現の問題
- 抽象的すぎる
- 例が不適切
- 文章が長すぎる

---

### ステップ3: 修正案の作成

**原因に応じた修正方針**:

#### A. 説明不足 → 詳細化
```markdown
❌ 修正前:
Histogramが拡大すればトレンド強化

✅ 修正後:
Histogram = MACD線とSignal線の差

Histogram拡大 = 差が広がる = トレンドが強くなっている
例: Histogram +0.01 → +0.05 = 上昇トレンド加速中
```

#### B. 前提知識の欠如 → 前提を追加
```markdown
✅ 追加:
「このセクションを理解するには、Week 0で学んだ『移動平均』の知識が必要です。
Week 0を復習してから読んでください」

または

✅ 簡単な復習を追加:
「移動平均 = 過去N日間の平均価格。トレンドを滑らかにする」
```

#### C. 構成の問題 → 再構成
```markdown
✅ 修正:
1. 概念の説明
2. なぜ重要か
3. 具体例
4. 実践での使い方

の順番に並べ替える
```

#### D. 表現の問題 → 対話形式に変換
```markdown
❌ 修正前（一方的な説明）:
Histogramは拡大すればトレンド強化、縮小すればトレンド弱化を示します。

✅ 修正後（対話形式）:
**サヤ**: 「Histogramが大きくなるってどういうこと？」
**ミコ**: 「MACD線とSignal線の差が広がるってことだ」
**サヤ**: 「差が広がる = トレンドが強い？」
**ミコ**: 「そう！勢いが増してる証拠だ」
```

---

### ステップ4: 修正の実施

**修正手順**:
1. 該当箇所を特定
2. 修正案を適用
3. 前後の文脈を確認（整合性チェック）
4. 他のWeekにも同様の問題がないか確認

## 💡 よくある問題と対策

### 問題1: 「専門用語が多すぎる」

**対策**:
- 初出時に必ず言い換える
- 用語集ページを作る
- 対話の中で「難しい言葉」と認識させる

**例**:
```markdown
**サヤ**: 「『相対力指数』って難しい...」
**ミコ**: 「確かに。『買われすぎメーター』の方がわかりやすいな」
```

### 問題2: 「なぜそうなるのかわからない」

**対策**:
- 「なぜ？」を3回繰り返す
- 因果関係を明示する
- 反例を示す

**例**:
```markdown
なぜRSI 70以上が危険？
→ なぜ？ → みんな買った
→ なぜ危険？ → もう買う人がいない
→ なぜ価格が下がる？ → 買う人がいないから
```

### 問題3: 「抽象的すぎる」

**対策**:
- 具体例を必ず入れる

## 🎯 実践例：00_foundations_interactive.md の改善

### 現在の問題点

1. **キャラクター設定が薄い**
   - サヤの背景が不明
   - なぜ投資を始めたいのか不明
   - 動機が弱い

2. **ストーリー性が薄い**
   - 場所・時間の描写がない
   - 感情の動きが少ない
   - 対話が堅い

3. **一度に全部書いたため、後半が簡略化**

### 改善案

**方針**: 骨子を作り直して、段階的に埋めていく

**手順**:
1. 骨子を作成（Scene数、各Sceneの目的を明確化）
2. 登場人物を詳細化
3. Scene 1-5を詳細化
4. Scene 6-10を詳細化
5. Scene 11-15を詳細化
6. レビュー・洗練

**注意**: 一度に全部書き直さない。段階的に改善。
