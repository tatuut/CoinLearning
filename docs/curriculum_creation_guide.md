# 📚 カリキュラム作成ガイド

## 🎯 目的

質の高い教材を段階的に作成するための標準手順書

本ガイドでは、2つの教材形式をカバーします：

### 📖 Week形式（実践的ツール習得教材）
**対象**: 100円→1000円を実際に達成したい実践者
**内容**: ツールの使い方、具体的な手順、トラブルシューティング
**目標文字数**: 1つのWeekあたり10,000文字以上

### 📚 Chapter形式（技術習得ストーリー）
**対象**: 技術の「なぜ」「どうやって」を深く理解したい学習者
**内容**: 技術の発明背景、数式の意味、実装方法、使い方
**目標文字数**: 1つのChapterあたり5,000-8,000文字

**方針**: どちらも一気に書かず、段階的に作成することで質を担保

**詳細サンプル**:
- Week形式の詳細例: [samples/week_format_detailed_example.md](./samples/week_format_detailed_example.md)
- Chapter形式の詳細例: [samples/chapter_format_detailed_example.md](./samples/chapter_format_detailed_example.md)

# Week形式の作成手順

## 📋 作成の5ステップ

### ステップ1: 概要と最終ゴール策定

**目的**: このWeekで何を学ぶか、何ができるようになるかを明確化

**作成内容**:
```markdown
# Week X: タイトル

## このWeekの概要
- 学ぶこと
- 達成目標
- 前提知識

## 最終ゴール
- このWeekが終わった時点でできること
- 使えるようになるツール
- 身につく知識・スキル

## 登場人物の成長
- ユウタの現状（Week開始時）
- ユウタの到達点（Week終了時）
- ミコが教えること
```

**ファイル保存**: `curriculum/weekX_<topic>_outline.md`

**チェックポイント**:
- [ ] Week全体の流れが明確か
- [ ] 前のWeekからの繋がりが自然か
- [ ] 最終ゴールが具体的か
- [ ] 使用するツールがリストアップされているか

### ステップ2: 詳細な骨子を作成

**目的**: 章立てと各セクションの内容を構造化

**作成内容**:
```markdown
## 骨子

### Part 1: ストーリー導入（1000文字目安）
- Scene 1: Week開始時の状況
- Scene 2: 新しい課題の発見
- Scene 3: ミコの提案

### Part 2: ツール・概念の説明（2000文字目安）
- 2.1 なぜこのツールが必要か
- 2.2 ツールの概要
- 2.3 基本的な使い方

### Part 3: 実践パート1（2000文字目安）
- 3.1 ユウタが実際に試す
- 3.2 最初の失敗
- 3.3 原因分析

### Part 4: 実践パート2（2000文字目安）
- 4.1 改善して再挑戦
- 4.2 ツールを使いこなす
- 4.3 成功体験

### Part 5: あなたの実践ガイド（2000文字目安）
- 5.1 具体的手順（コマンド例付き）
- 5.2 トラブルシューティング
- 5.3 チェックリスト

### Part 6: まとめと振り返り（1000文字目安）
- 6.1 学んだことリスト
- 6.2 次のWeekへの予告
- 6.3 ユウタからのメッセージ
```

**ファイル保存**: 同じファイルに追記

**チェックポイント**:
- [ ] 各パートの文字数目安が適切か（合計10000文字）
- [ ] ストーリーと実践のバランスが取れているか
- [ ] 登場人物の行動に具体的な手順が含まれているか

## 💡 Tips（共通）

### コミットメッセージの書き方

#### Week形式の場合
```
Week X: ステップY-Z完了 - 内容の概要

【このコミットで追加した内容】
- ...
- ...

【次のステップ】
- ...
```

**例**:
```
Week 2: ステップ3-5完了 - Part 5（あなたの実践ガイド）

【このコミットで追加した内容】
- 手順1-5の詳細な実行手順
- トラブルシューティング10項目
- 実践前後のチェックリスト

【次のステップ】
- Part 6を執筆（まとめと振り返り）
```

#### Chapter形式の場合
```
Chapter X: Scene Y完了 - シーンの概要

【このコミットで追加した内容】
- ...

【次のシーン】
- ...
```

**例**:
```
Chapter 2: Scene 3完了 - 数式の誕生

【このコミットで追加した内容】
- ワイルダーの思考プロセス（Step 1-4）
- RSI数式の導出過程
- バックテストによる検証シーン

【次のシーン】
- Scene 4: 1978年6月、伝説の出版
```

### 文字数の確認方法

```bash
# Linuxの場合
wc -m week2_technicals.md

# Windowsの場合
powershell -command "(Get-Content week2_technicals.md | Measure-Object -Character).Characters"
```

### Claude Codeとの協働

#### Week形式の場合

1. **一つのステップごとに依頼**
   - 「Week 2のステップ1をお願いします」
   - 「Part 3を埋めてください」

2. **フィードバックループ**
   - 各ステップ完了後にレビュー
   - 修正が必要なら指摘
   - OKならコミットして次へ

3. **並列作業は避ける**
   - 1つずつ順番に進める
   - コミット履歴を綺麗に保つ

#### Chapter形式の場合

1. **調査から依頼**
   - 「Chapter 2（RSI）の発明者について調査してください」
   - 「ワイルダーの背景、1978年の時代背景、RSI誕生の経緯を調べて」

2. **骨子レビュー**
   - 12シーン構成を確認
   - ストーリーの流れが自然か
   - 技術的な正確性をチェック

3. **シーンごとに執筆**
   - 「Scene 1-2を執筆してください」
   - 「Scene 3の数式誕生シーンを書いてください」
   - 各シーンごとにコミット

4. **技術的正確性の確保**
   - 数式が正しいか確認
   - 歴史的事実が正確か確認
   - WebSearchで最新情報を補完

# Chapter形式の作成手順

## 🎯 Chapter形式とは

技術が発明された背景、数式の意味、実装方法を**物語形式**で学ぶ教材。

**例**: RSI（Relative Strength Index）を学ぶ場合
- ❌ 「RSIは70以上で買われすぎ、30以下で売られすぎを示す指標です」（暗記型）
- ✅ 「1978年、機械エンジニア出身のワイルダーは、何度も負けて悩んだ末、『エンジンの正常範囲』という概念を株価に応用し、RSIを発明した」（理解型）

**参考**: [実際のChapter例](./stories/02_rsi_invention.md)、[詳細サンプル](./samples/chapter_format_detailed_example.md#L1-L50)

## 📋 Chapter形式の6ステップ

### ステップ1: 発明者と技術を調査

**目的**: 歴史的事実と技術の本質を理解

**調査項目**:
- 発明者の背景（職業、年齢、動機）
- 発明された時代背景（いつ、どこで）
- 解決したかった問題
- 数式・アルゴリズムの意味
- 現代での使われ方

**情報源**:
- Wikipedia、学術論文、原典書籍
- WebSearch ツールで最新情報を確認

**成果物**: `調査メモ.md`（発明者プロフィール、年表、数式、参考文献）

**詳細**: [Chapter形式サンプル - 調査段階](./samples/chapter_format_detailed_example.md#L11-L30)

### ステップ3: ストーリーパート執筆（Scene 1-5）

**目的**: 発明者の物語を臨場感を持って描く

**重要ポイント**:
- 具体的な時代・場所（例: 1978年、ニューオーリンズ）
- 発明者の失敗体験（読者が共感できる）
- 専門知識の転用（例: エンジン工学 → 株価分析）
- 数式誕生の必然性（Step 1 → Step 2 → ... → 完成）

**執筆方法**: Scene 1 → コミット → Scene 2 → コミット ... と順次進める

**詳細**: [Chapter形式サンプル - Scene 1-5](./samples/chapter_format_detailed_example.md#L50-L200)

### ステップ4: 実装パート執筆（Scene 6-9）

**目的**: 理論を実践コードに落とし込む

**Scene 6: Pythonでの実装**
- シンプルな関数
- コメントで各ステップを説明（Scene 3の数式と対応）

**Scene 7: 実データでの検証**
- Week 1の失敗例など、既存のデータで検証
- 数値と解釈をセットで

**Scene 8: 応用的な使い方**
- 3つの使い方を提示
- 初心者へのアドバイス（「最初は基本だけ」）

**Scene 9: 限界と弱点**
- 正直に弱点を伝える
- どんな場面で失敗するか
- 解決策（他指標との組み合わせ）

# Week形式とChapter形式の使い分け

## 📖 学習パターン

### パターン1: 実践優先型（初心者向け）
1. Week 1 → 実践で100円→104円を体験
2. Chapter 1-2 → なぜ成功/失敗したか理解
3. Week 2 → 学んだ技術を使って実践
4. Chapter 3-4 → さらに深く理解

**メリット**: 成功体験が先にあるので、モチベーション維持しやすい

### パターン2: 理解優先型（中級者向け）
1. Chapter 1-6 → 技術を一通り学ぶ
2. Week 1-4 → 実践で技術を使いこなす

**メリット**: 理論武装してから実践するので、失敗が少ない

### パターン3: ハイブリッド型（推奨）
- Week 1（実践） → Chapter 1-2（理解） → Week 2（実践） → Chapter 3-4（理解） ...

**メリット**: 実践と理解を交互に繰り返すので、定着しやすい

---

## 🔄 作成優先度

### 最初に作るもの
1. **Week 1-4**: 実践教材を先に完成させる
   - 読者がすぐに使えるツールと手順
   - 100円→1000円の道筋を示す

### 並行して作るもの
2. **Chapter 1-6**: Weekで使う技術の背景を説明
   - Week 1 完成後、Chapter 1-2 を作成
   - Week 2 完成後、Chapter 3-4 を作成

### 相互リンク戦略
- Week内で「詳しくは Chapter X を参照」とリンク
- Chapter内で「実践は Week X を参照」とリンク

**例**:
```markdown
## Week 2: テクニカル分析の実践

**ミコ**: 「RSIを使うぞ」

**ユウタ**: 「RSIって？」

**ミコ**: 「詳しくは [Chapter 2: RSIの発明](../docs/stories/02_rsi_invention.md) を読め。
ここでは使い方だけ教える」
```

## 💡 Tips

### Week形式のTips

✅ **超詳細な手順**
- 「コマンドを実行」ではなく「以下のコマンドをコピーして、コマンドプロンプトに貼り付け、Enterを押す」
- スクリーンショット（テキストで説明）を多用

✅ **失敗→改善のサイクル**
- Part 3で失敗、Part 4で成功
- 失敗の原因を明確に（「ニュースを無視した」「ボラティリティが高すぎた」）

✅ **チェックリスト必須**
- 実践前、実践中、実践後でチェック
- 読者が自分で進捗確認できる

**詳細**: [Week形式詳細サンプル](./samples/week_format_detailed_example.md)

---

### Chapter形式のTips

✅ **ストーリーの臨場感**
- 「1978年、ニューオーリンズの書斎で...」と具体的に
- セリフと地の文のバランス（セリフ60%, 地の文40%）

✅ **数式の必然性**
- 「Step 1で○○を試した → ダメだった → Step 2で改善 → ...」
- 各Stepの理由を明確に

✅ **正直さ**
- 弱点を隠さない（「強いトレンド相場では役に立たない」）
- 万能ではないことを伝える

**詳細**: [Chapter形式詳細サンプル](./samples/chapter_format_detailed_example.md#L680-L750)
